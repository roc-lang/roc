# Findings

1. Monotype specialization still works by cloning and mutating checker-style solved stores instead of consuming explicit monotype specialization artifacts. `src/monotype/lower.zig:265` clones each published typed-CIR module’s `types.Store` and `Ident.Store`, `src/monotype/lower.zig:9338` still runs checker unification against that mutable clone, and `src/monotype/lower.zig:9457` still copies source vars across module stores on demand. This is no longer a checker-boundary leak, but it is still architectural residue: monotype specialization is reusing HM solved-store machinery as its workspace rather than using explicit specialization facts keyed by source identity and specialization key.

2. Monotype still re-derives static-dispatch targets from receiver type shape instead of consuming a fully explicit earlier fact. `src/monotype/lower.zig:9577` starts target resolution during lowering, `src/monotype/lower.zig:9626` looks methods up in the target module table, and `src/monotype/lower.zig:9712` walks aliases/nominals in the receiver var to infer the target type/module. That is after-the-fact analysis of solved type structure; the checker should publish the resolved dispatch target directly.
